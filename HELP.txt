================================================================================
                     RUBIX PROTOCOL — USER GUIDE
================================================================================

AI development agent with persistent memory, self-healing, and multi-instance
orchestration. Built as an MCP server for Claude Code.

160 tools. 246 TypeScript files. 35+ subsystems.

================================================================================
                           QUICK START
================================================================================

1. Install and build:
     git clone https://github.com/BobbyRuby/rubix-protocol.git
     cd rubix-protocol
     npm install && npm run build

2. Add to your project's .claude/mcp.json:
     {
       "mcpServers": {
         "rubix": {
           "command": "node",
           "args": ["dist/mcp-server.js"],
           "cwd": "/path/to/rubix-protocol",
           "env": {
             "OPENAI_API_KEY": "sk-...",
             "ANTHROPIC_API_KEY": "sk-ant-...",
             "RUBIX_DATA_DIR": "./data"
           }
         }
       }
     }

3. Restart Claude Code after adding config.

4. Use god_codex_do({ description: "your task", codebase: "." }) to submit a task.


================================================================================
                        DEPLOYMENT MODES
================================================================================

MCP Server (primary)
--------------------
Runs as a tool server inside Claude Code via .claude/mcp.json.
  npm run start:mcp

Standalone Services
-------------------
Run as a standalone service with Telegram bot, HTTP webhooks, and scheduling:
  npm run launch              # All services
  npm run launch:telegram     # Telegram bot only
  npm run launch:daemon       # Daemon + webhooks
  npm run launch:webhooks     # Webhook server only (port 3456)

Multi-Instance Orchestra (tmux)
-------------------------------
Run multiple Claude Code instances coordinating via shared message bus:
  bash scripts/rubix-orchestra.sh start 3    # Launch 3 instances + monitor
  bash scripts/rubix-orchestra.sh status     # Check heartbeats + queue
  bash scripts/rubix-orchestra.sh attach     # Reattach to tmux session
  bash scripts/rubix-orchestra.sh stop       # Shutdown

Instances communicate via comms.db — persistent SQLite message bus with
heartbeats, threading, priority routing, broadcast, and inter-instance
permission relay. Worker permission requests route to the orchestrator
automatically via the permission hook.

================================================================================
                     CODEX EXECUTION PIPELINE
================================================================================

When you submit a task via god_codex_do, CODEX runs a 6-phase pipeline:

  Phase 1: CONTEXT SCOUT (Sonnet)  — gather codebase context, detect skills
  Phase 2: ARCHITECT (Opus)        — design solution, assess complexity
  Phase 3: ENGINEER (routed)       — write code (parallel for high-complexity)
  Phase 4: CODE REVIEW             — OWASP security scan, guardian audit
  Phase 5: VALIDATOR               — test, lint, type-check
  Phase 6: FIX LOOP                — 5-tier self-healing escalation:
           Sonnet → Sonnet(alt) → Sonnet+think → Opus → Opus+think

Model routing: low complexity → Haiku, medium → Sonnet, high → Opus.


================================================================================
                        MEMORY SYSTEM
================================================================================

STORE:  content → L-Score provenance → SQLite + embed → vector index
QUERY:  text → embed → vector search → MemRL two-phase ranking → results
LEARN:  SONA trajectory feedback → EWC regularization → auto-prune/boost
CAUSAL: hypergraph relations (causes|enables|prevents|correlates|precedes|triggers)

Memory types: component, department, mcp_tool, capability, workflow, config,
error_pattern, success_pattern, bug_fix, dev_feature, arch_insight, system, generic

Query tools:
  god_query              Semantic vector search
  god_query_expanded     Human-readable decoded output
  god_self_query         Query system self-knowledge


================================================================================
                   INTER-INSTANCE COMMUNICATION
================================================================================

Multiple instances coordinate via comms.db (SQLite message bus):

  god_comms_heartbeat    Register presence (required first)
  god_comms_send         Send to specific instance
  god_comms_broadcast    Send to all instances
  god_comms_inbox        Check unread messages
  god_comms_read         Read message content
  god_comms_ack          Mark as processed
  god_comms_thread       Get conversation thread
  god_comms_peers        List known instances
  god_comms_escalate     Auto-routed question (Telegram or CLI fallback)

Permission relay: worker permission requests automatically route to the
orchestrator (instance_1) via comms.db when running in orchestra mode.


================================================================================
                        AFK MODE
================================================================================

When AFK is active, everything routes through Telegram:
  - Tool permission requests → Telegram Allow/Deny buttons
  - Questions and escalations → Telegram
  - Notifications → Telegram

Toggle via:
  Telegram: /afk command
  MCP:      god_afk action="toggle"
  API:      POST http://localhost:3456/api/afk

================================================================================
                       MULTI-PROJECT SETUP
================================================================================

Each project gets its own isolated MCP instance with separate memory,
embeddings, task queues, and containment rules.

In .claude/mcp.json:
  {
    "mcpServers": {
      "rubix-backend": {
        "command": "node",
        "args": ["dist/mcp-server.js"],
        "cwd": "/path/to/rubix-protocol",
        "env": {
          "RUBIX_DATA_DIR": "./data/projects/backend",
          "RUBIX_PROJECT_ROOT": "/path/to/backend-api",
          "RUBIX_PROJECT_NAME": "Backend API"
        }
      }
    }
  }

Optional shared knowledge: set RUBIX_CORE_BRAIN_DATA_DIR to a shared data
directory for cross-project learning.

================================================================================
                       TROUBLESHOOTING
================================================================================

"OPENAI_API_KEY environment variable is required"
  → Set OPENAI_API_KEY in .claude/mcp.json env block

"ANTHROPIC_API_KEY not set"
  → CODEX won't generate code without it

"SqliteError: no such column"
  → Schema mismatch. Delete ./data/memory.db and restart.

Telegram bot not responding:
  → Check TELEGRAM_BOT_TOKEN is correct
  → Only one instance can poll at a time

Webhook server won't start:
  → Port 3456 in use. Set WEBHOOK_PORT to another port.

MCP tools not appearing:
  → Restart Claude Code after editing .claude/mcp.json
  → Verify the cwd path to rubix-protocol is correct
